## 1. Backend / service impact

No backend, no service, no daemon. The feature is a pure-bash verb in `lib/verbs/agents_md.sh`
(or an equivalently named file — underscore preferred to match filesystem conventions). It reads
`claude/agents/massoh-*.md` from the Massoh install root (`$MASSOH_HOME`) using bash glob, parses
YAML frontmatter fields (`name:`, `description:`), synthesizes a markdown summary table, and writes
`AGENTS.md` at the repo root. The only runtime dependency is the installed Massoh role files — no
network, no LLM, no subprocess beyond `grep`/`sed`/`awk`.

the task request uses both) is additive. No existing invocation changes. The generated `AGENTS.md`
is an output artifact in the user's repo; it is opt-in (the user must run the verb). No existing
user file is altered unless it carries the generated sentinel.

## 6. Safety / guardrail risks and conditions

### AM1 — `|| true` on every read operation; degrade exit 0 on missing role files

The sourcing glob `"$MASSOH_HOME/claude/agents/massoh-*.md"` may expand to nothing if Massoh is
not installed or the path is wrong. Under `set -euo pipefail`, a glob with no matches in a `for`
loop is safe (loop body never executes); but downstream `grep`/`awk`/`sed` operations against
individual files must all be guarded with `|| true`. If no role files are found, the verb must
print a human-readable message (e.g. "massoh agents-md: no role files found at $MASSOH_HOME/
claude/agents/ — is Massoh installed?") and exit 0. It must NOT exit 1 in the degrade path, as
doing so would break any wrapper that calls the verb speculatively.

Reviewer check: `grep -v '#' lib/verbs/agents_md.sh | grep -E '(grep|awk|sed|head|tail)' | grep -v '|| true'`
must return empty.

### AM2 — Write location + clobber policy (sentinel guard; mirror BR5)

Write target: `AGENTS.md` in the current repo root (`$repo`, resolved via
`git -C "$PWD" rev-parse --show-toplevel 2>/dev/null || echo "$PWD"`).

Clobber policy (exact mirror of BR5 from TASK-2026-06-19-board-renderer):
- If `AGENTS.md` does NOT exist: create it (write normally).
- If `AGENTS.md` exists AND contains the line `<!-- massoh-generated -->` (the generator
  sentinel): overwrite it. The sentinel must be written as the first line of the generated file
  on every run so subsequent runs detect it and overwrite safely.
- If `AGENTS.md` exists AND does NOT contain `<!-- massoh-generated -->` (i.e. it was
  hand-authored by the user or another tool): print an error message to stderr and exit 1
  without modifying the file. Suggested message:
  "massoh agents-md: AGENTS.md exists but was not generated by Massoh (no sentinel). Refusing
  to overwrite. Delete it or add '<!-- massoh-generated -->' to allow overwrite."

The sentinel line must be `<!-- massoh-generated -->` (consistent with the board-renderer
precedent for HTML output) — a single exact string, not a variant. The file structure is:

```
<!-- massoh-generated -->
# Massoh Agent Team

...table...

...workflow pointer...
```

Idempotency proof: running the verb twice on a repo produces byte-identical `AGENTS.md` (given
unchanged role files). The reviewer must assert this via a diff of two consecutive runs.

### AM3 — Read role frontmatter as DATA only; no source/eval; cap field lengths

The verb must parse `name:` and `description:` from each role file's YAML frontmatter block
(the text between `---` and the next `---`). The parse must:
- Use `grep`/`sed`/`awk` — no `source`, no `eval`, no `bash -c` containing file content.
- Cap `name` at 64 characters and `description` at 256 characters (truncate silently with a
  trailing `...` if needed). This prevents a maliciously long description from producing a
  multi-megabyte AGENTS.md.
- Read only the first occurrence of `name:` and `description:` per file (use `head -n1` on
  grep output).
- Derive `edits code?` from the frontmatter `tools:` field: if `tools:` contains `Edit` or
  `Write`, emit `yes`; otherwise emit `no`. This is a static read of the tools list, not an
  inference.

Reviewer check: `grep -E 'source |eval |bash -c' lib/verbs/agents_md.sh` (non-comment lines)
must return empty.

### AM4 — Sanitize field values into markdown table cells (pipe + newline injection)

Any `name:` or `description:` value that contains a `|` character must have the pipe replaced
with a safe substitute (e.g. `/` or the HTML entity `&#124;`) before being written into the
markdown table. Any embedded newline in a frontmatter value (from multi-line YAML strings)
must be collapsed to a single space. The `edits code?` column value is always `yes` or `no`
(no user-controlled input flows there unguarded).

This is the same class of sanitization as BR3 (BOARD.md cell sanitization precedent). A single
sanitize helper function (e.g. `_agents_md_sanitize_cell`) must be used for both `name` and
`description` columns so the logic is in one place and testable.

### AM5 — Scope guard: SUMMARY index only; do NOT dump full role body content

The generated `AGENTS.md` must contain:
1. The generator sentinel comment (`<!-- massoh-generated -->`).
2. A one-line header (e.g. `# Massoh Agent Team`).
3. A markdown table with columns: `Role` | `Description` | `Edits code?` — one row per role
   file found (7 rows for a standard install).
4. A section pointer to the gated workflow, e.g.:
   `> Workflow: see [agent-os/policies/03_AGENT_WORKFLOW.md](agent-os/policies/03_AGENT_WORKFLOW.md)`
   and optionally a pointer to `README.md` for the full role table.
5. Nothing else — no role body content, no policy text, no full CLAUDE.md preamble, no
   multi-paragraph role descriptions. This is a pointer document.

The reviewer must confirm by inspection that no role body content (beyond the frontmatter
`description:` field, capped at 256 chars) appears in the generated file.

### AM6 — `set -euo pipefail`; all compound conditions use `if/else`, not `A && B || C`

The new verb file must open with `set -euo pipefail` (or inherit it from bin/massoh's bootstrap,
which already sets it). Any compound boolean that has side-effects must use explicit `if/else`,
not the `A && B || C` anti-pattern (the cadence-cron REQUEST CHANGES lesson, recorded in
AGENT_SYNC.md). Reviewer check: grep for `&&.*||` patterns in non-trivial statement positions.

### AM7 — Additive dispatch: exactly 1 `case` branch + 1 usage token in `bin/massoh`

The implementer adds to `bin/massoh`:
- 1 `case` branch (e.g. `agents-md) shift || true; cmd_agents_md "$@" ;;`)
- 1 token in the usage `die` fallback string (e.g. appending `agents-md` to the verbs list)

No other change to `bin/massoh` is permitted. The reviewer must confirm via
`git diff HEAD -- bin/massoh | grep '^+' | grep -v '^+++' | wc -l` equals 2 (one line per
addition). If the function is inlined rather than file-sourced this count will differ — in that
case the reviewer must confirm no safety-critical function in `bin/massoh` is altered.

Preferred implementation: `lib/verbs/agents_md.sh` (new file), sourced by the existing glob
`"$MASSOH_HOME/lib/verbs/"*.sh`. The glob already covers any new `.sh` file in that directory
(confirmed by bin/massoh lines 172–176); no manifest.yml change is needed for the verb file.

### AM8 — No secrets, no network, no LLM

The verb must not:
- Read or log any environment variable matching `*TOKEN*`, `*KEY*`, `*SECRET*`, `*PASSWORD*`,
  or `*CREDENTIAL*`.
- Call `curl`, `wget`, `nc`, `ssh`, `gh`, or any network primitive.
- Call any LLM API or agent tool.

Reviewer check: `grep -iE 'curl|wget|nc |ssh |gh |TOKEN|SECRET|KEY|PASSWORD|CREDENTIAL' lib/verbs/agents_md.sh`
(non-comment lines) must return empty.

## 7. Expansion / localization risks

None. `AGENTS.md` is plain UTF-8 markdown. Role names and descriptions are static English strings
from the Massoh role files (themselves authored in English). No locale-specific formatting, no
RTL requirement, no region-specific content. The feature is itself expansion-neutral: it reads
whatever role files are installed, so a future domain pack (e.g. a Japanese-language role) would
appear automatically if its frontmatter uses UTF-8 `name:` and `description:` fields.

No region/locale/segment hard-coding is introduced. The expansion principle
(12_EXPANSION_READY_ARCHITECTURE.md) is satisfied.
